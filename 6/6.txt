Задача №6.

Легенда.
Вы участвуете в проекте внедрения робота-сортировщика на складе e-commerce компании.

Робот:
1. получает задания через API
2. возвращает статусы выполнения
3. интегрируется с WMS системой заказчика

Есть проблемы:
1. иногда задания «теряются»
2. статусы приходят с задержкой
3. в логах сложно понять причину ошибок

Ситуация:
- заказчик жалуется: робот «теряет задания»
- разработчики говорят: «у нас всё ок»
- логи есть, но они messy

Вопрос:
Что вы будете делать? Опишите пошагово.

------------------------------------------------------------

1. Берём конкретные кейсы:
- конкретные tasks 
- временной диапазон
- ожидаемый и фактический результаты


2. Есть цепочка (flow) движения задания в сети: 
   WMS -> API -> очередь/шина -> робот -> статус -> API -> WMS.

   Выясняем, где именно потерялось задание, например: 
   WMS не отправил / API принял, но не передал / очередь потеряла / робот не получил / робот выполнил, но статус не ушёл /статус ушёл, но WMS не принял

   Смотрим по логам (ранее говорилось, что у каждого звена в этой цепочке свои логи)


3. Используем трассировку (correlation ID)
   Связь разрозненных записей в логах одного бизнес-процесса сводим в единую цепочку.


4. Нужен retry (механизм повторного запуска при отсутствии подтверждения о доставке)
   Можно написать свой велосипед, можно взять готовые решения, например подтверждение доставки (ACK — Acknowledgment) есть в TCP. 


5. Задержки логов 
   Задержки это не пропажа данных, но тоже можно посмотреть, т.к. статус может придти позже истечения timeout


6. Нормализация логирования 
   - Ввести структурированные логи (перевести их в формат JSON)
   - Очистка от мусора (лишние поля) 
   - и ещё ряд возможных действий (первых двух пунктов должно хватить)


7. Ввести опрос ключевых показателей
   Регулярно (например раз в час) генерятся показатели 'количество отправленных задач',
   'количество полученных роботом', 'количество выполненных' и на выходе итоговая метрика 'retry count'.
   В идеале ожидаем 'количество отправленных == количество полученных'.

8. Провести несколько тестовых испытаний с разным числом заданий.

9.  В итоге понимаем тип проблемы: 
    - нет перезапуска при отсутствии подтвеждения о доставке
    - дубликаты ID, 
    - нагрузочная (переполнение)
    - проблемы железа
    - тайминговая (статус приходит позже timeout)
    - что-то ещё

10. Исправляем
