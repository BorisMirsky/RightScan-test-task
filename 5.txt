1. Промпт для анализа логов

```
Имеются логи робота-сортировщика за последние 'n' часов. 
В системе теряются задания (нет финального статуса) и статусы приходят с задержкой более 30 секунд.

Проанализируй приложенный файл с логами. Выполни следующие действия:

Сгруппируй ошибки по типу (превышение таймаута, ошибки сериализации, проблемы сети, ошибки WMS).

Выдели id заданий, которые начались, но не завершились. Составь таймлайн для самых подозрительных кейсов.

Определи, есть ли корреляции между типом товара и частотой ошибок. 

Предложи гипотезы, почему задания теряются.

ВЫВЕДИ:
- Таблицу с ошибками (timestamp | сообщение | компонент | частота)
- Гипотезы о причинах 
- Рекомендации для воспроизведения бага
- Предложения по мониторингу (в частности, какие алерты настроить)
```


2. Промпт для генерации документации

```
Напиши документацию для робота-сортировщика, который интегрируется с WMS.
Учти, что прямо сейчас есть проблемы, связанные с потерей заданий и задержками статусов.

Создай документ, содержащий:

- Спецификацию API (OpenAPI/Swagger style): эндпоинты /task (POST) и /status (PUT/GET). Укажи требования к идемпотентности (x-request-id) и таймаутам.

- Схему последовательности действий, показывающую обмен данными между WMS, роботом и очередями сообщений. 

- Справочник статусов: что означает каждый статус (PENDING, IN_PROGRESS, COMPLETED, LOST, ERROR) и при каких условиях робот переходит в состояние ERROR.

- Рекомендации по обработке ошибок.

- Раздел "Как отладить интеграцию": список полей в логах (trace_id, robot_id, wms_order_id), которые необходимо логировать для сквозной трассировки.
```


3.  Промпт для поиска причин багов

```
В системе есть проблемы: теряются задания (статус не доходит до WMS) и статусы приходят с задержкой до нескольких минут.

Изучи код модуля сортировки и обработки заданий. Предположи, где причины ошибок:

- Есть ли гонка данных (race conditions) при обновлении состояния задания в памяти или базе данных? 

- Может ли задание быть затерто новым?

- Проверь реализацию HTTP клиента. Правильно ли настроены таймауты? Обрабатываются ли полуоткрытые соединения? Как ведет себя consumer при ошибке коммита offset в RabbitMQ?

- Управление ресурсами: Нет ли утечки горутин/потоков при долгих операциях? Используются ли context.WithTimeout везде, где это необходимо?

- Логика повторных попыток: Не приводит ли ретрай на стороне робота к дублированию физических действий (повторной сортировке того же товара)?

- Напиши фрагменты кода, которые гарантируют "at-least-once" доставку статуса без потери данных, даже при падении робота.
```
