


1. Схема работы цепочки WMS → API → робот → API → WMS в виде текста


WMS формирует задание (Например: переместить коробку из ячейки A в ячейку B)
WMS присваивает task_id
WMS отправляет задание в API робота с параметрами: POST /tasks(task_id, тип операции, координаты, приоритет)
API принимает задание
API валидирует данные
API сохраняет в БД
API ставит статус NEW
API кладёт в очередь роботов
Робот запрашивает задания: GET /tasks/next?robot_id=R1
API выбирает подходящее задание -> статус меняется на ASSIGNED
Робот начинает выполнение
Робот отправляет статус IN_PROGRESS
API обновляет БД
Робот выполняет действие
Робот перемещает товар
Робот сканирует штрихкод
Робот подтверждает выполнение
Робот отправляет финальный статус DONE или FAILED
API принимает статус
API обновляет запись
API отправляет результат в WMS
WMS получает результат
WMS обновляет состояние заказа
WMS логирует выполнение

3. Возможные статусы задания 

Полный список статусов:
NEW — создано WMS
QUEUED — в очереди на выполнение
ASSIGNED — назначено роботу
IN_PROGRESS — робот выполняет
DONE — успешно выполнено
FAILED — ошибка выполнения
CANCELLED — отменено WMS
TIMEOUT — робот не ответил
RETRY — повторная попытка
LOST — задание потеряно


То же самое, но на конкретном примере Wildberris 

                     От отгрузки до размещения на хранение:
«Отгружено на воротах» (прибытие),                  
«Идёт приёмка» (поштучный пересчет), 
«Принято» (товар готов к продаже),                 Товар появляется в домене Склад 
«Принято с расхождениями» (обнаружены ошибки).
                     От оформления до получения: 
«Оформлен» (принят), 
«Отправлен на сборку» (комплектация), 
«Собран» (упакован), 
«Отгружен» (передан логистам),                      Товар покидает домен Склад 
«В пути» (едет), 
«В пути на ПВЗ», 
«Готов к выдаче» (можно забирать), 
«Доставлен» (выдан), 
«Задерживается» (перегрузка).

4. Причины потери заданий 

--- Сетевые проблемы
WMS отправил → API не получил
API ответил → WMS не получил ACK
робот получил → статус не отправился

--- Нет идемпотентности
дублирующиеся task_id
перезапись задания
race condition

--- Проблемы очереди
сообщение упало из очереди
TTL истёк
очередь переполнена

--- Ошибка транзакций БД
задание выдано роботу, но commit не прошёл
статус обновился частично

--- Робот "забрал" задание, но умер
power off
потеря Wi-Fi
reboot

--- Нет heartbeat робота
система думает, что робот выполняет
робот уже недоступен

--- Некорректная логика retry
задача упала, но не вернулась в очередь
задача застряла в IN_PROGRESS

--- Проблемы callback в WMS
API отправил результат
WMS не принял
нет retry отправки
Как обычно решают такие проблемы 
idempotency key
retry с exponential backoff
heartbeat роботов
watchdog timeout
DLQ (dead letter queue)
event log
correlation_id
distributed tracing
